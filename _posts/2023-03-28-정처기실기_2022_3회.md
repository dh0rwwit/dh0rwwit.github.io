관계대수는 관계형 데이터베이스에서 원하는 정보와 그 정보를 검색하기 위해 어떻게 유도하는가를 기술하는 절차적인 언어이다.<br>
관계대수에 사용되는 연산은 다음과 같다. <br>
    - 합집합 UNION은 두 릴레이션에 존재하는 튜플의 합집합을 구하되, 결과로 생성된 릴레이션에서 중복되는 튜플을 제거되는 연산으로, 사용되는 기호는 ( 1 )이다. <br>
    - 차집합 DIFFERENCE는 두 릴레이션에 존재하는 튜플의 차집합을 구하는 연산으로 기호는 ( 2 )이다. <br>
    - 교차곱 CARTESIAN PRODUCT는 두 릴레이션에 있는 튜플들의 순서쌍을 구하는 연산으로 사용되는 기호는 ( 3 )이다. <br>
    - 프로젝트 PROJECT는 주어진 릴레이션에서 속성 리스트 Attribute List에 제시된 속성값만을 추출하여 새로운 릴레이션을 만드는 연산으로 사용되는 기호는 ( 4 )이다. <br>
    - 조인 Join은 공통 속성을 중심으로 두 개의 릴레이션을 하나로 합쳐서 새로운 릴레이션을 만드는 연산으로, 사용하는 기호는 ( 5 )이다. <br>

    3. × <br>
    4. π <br>
    5. ⋈ <br>

문제 3. 디자인패턴에 대한 설명에서 괄호에 들어갈 알맞은 용어를 찾아 쓰시오 <br>
- ( 1 )패턴은 구현부에서 추상층을 분리하여, 서로가 독립적으로 확장할 수 있도록 구성한 패턴으로 기능과 구현을 두개의 펼도 클래스로 구현한다는 특징이 있다. <br>
- ( 2 )패턴은 한 객체에서 상태가 변화하면 객체에 상속된 다른 객체들에게 변화된 상태를 전달하는 패턴으로, 일대다의 의존성을 정의한다. <br>
주로 분산된 시스템 간에 이벤트를 생성*발행하고 이를 수신해야할 때 이용한다. <br>

답 <br>
    1. Bridge <br>
    2. Observer <br>

;) 디자인 패턴 <br>
    1. 모듈 간 관계 및 인터페이스 설계할 때 참조할 수 있는 전형적인 해결 방식 또는 예제 <br>
    2. 생성패턴, 구조패턴, 행위패턴으로 구분된다. <br>

;) 인스턴스 <br>
    1. 클래스에 속한 각각의 객체. <br>

;) 생성패턴 <br>
1. 추상팩토리( Abstract Factory ) 
    - 인터페이스를 통해 서로 연관/의존하는 객체들의 그룹으로 생성하여 추상적으로 표현.  <br>
    - 연관된 서브 클래스들을 묶어 한번에 교체하는 것이 가능하다. <br>
2. 빌더( Builder )
    - 인스턴스를 건축하듯이 조합하여 객체를 생성함.  <br>
    - 생성과정과 표현방법을 분리하고 있어 동일한 객체 생성에서도 서로 다른 결과가 나온다. <br>
3. 팩토리 메소드( Factory Method ) 
    - 객체생성을 서브클래스에서 처리하도록 분리하여 캡슐화한 패턴 <br>
    - 상위클래스에서 인터페이스만 정의함. <br>
    - 가상 생성자( Virtual Constructor )패턴이라고도 함. <br>
4. 프로토타입( Prototype ) 
    - 원본 객체를 복제하는 방법으로 객체를 생성. <br>
    - 비용이 큰 경우 주로 이용함. <br>
5. 싱클톤( Singleton ) 
    - 하나의 객체 생성시, 생성된 객체 어디서든 참조할 수 있지만, 여러 프로세스가 동시에 참조할 수 없다. <br>
    - 클래스내에 인스턴스가 하나뿐임을 보장하며, 불필요한 낭비 최소화함. <br>

;) 구조패턴 <br>
1. 어댑터( Adapter ) 
    - 호환성이 없는 클래스들의 인터페이스들이 다른 클래스가 이용할 수 있게 변환해주는 패턴 <br>
2. 브리지( Bridge ) 
    - 구현부에서 추상층을 분리하여, 서로가 독립적으로 확장할 수 있도록 해주는 패턴. <br>
    - 기능과 구현을 2개의 별도 클래스로 구현함. <br>
3. 컴포지트( Composite ) 
    - 여러 객체를 가진 복합객체와 단일 객체를 구분없이 다루고자 할 때 사용하는 패턴. <br>
    - 객체들을 트리구조로 구성 -> 복합객체안에 복합객체가 포함되는 구조로 구현할 수 있다. <br>
4. 데코레이터( Decorator ) 
    - 객체 간 결합을 통해 능동적으로 기능들을 확장 가능. <br>
    - 임의의 객체에 부가적인 기능을 추가하기 위해 다른 객체를 덧붙이는 방식 이용. <br>
5. 퍼싸드( Facade ) 
    - 복잡한 서브 클래스들을 피해 더 상위에 인터페이스를 구성함. -> 서브 클래스들을 간편하게 사용할 수 있게 함. <br>
    - 서브 클래스들 사이에 통합 인터페이스를 제공하는 Wrapper가 필요함. <br>
6. 플라이웨이트( Flyweight ) 
    - 인스턴스를 공유하여 사용함으로써 메모리를 절약하는 패턴. <br>
    - 다수의 유사객체를 생성하거나 조작할 때 유용하게 사용할 수 있다. <br>
7. 프록시( Proxy ) 
    - 접근이 어려운 객체와 여기에 연결하려는 객체 사이에서 인터페이스 역할을 수행하는 패턴. <br>
    - 네트워크 연결, 메모리의 대용량 객체로의 접근 등에 주료 이용함. <br>

;) 행위패턴 <br>
1. 책임연쇄( Chain of Responsibility ) <br>
    - 요청을 처리하는 객체가 둘 이상 존재시, 한 객체가 처리하지 못하면 다음 객체로 넘어가는 패턴. <br>
    - 요청을 처리하는 객체들이 chain으로 묶여 있어 요청이 해결될 때까지 고리를 따라 책임이 넘어감. <br>
2. 커맨드( Command ) 
    - 요청을 객체형태로 캡슐화하여 재이용하거나 취소할 수 있도록 요청에 필요한정보를 저장하거나 로그에 남기는 패턴 <br>
3. 인터프리터( Interpreter ) 
    - 언어에 문법 표현을 정의하는 패턴. <br>
    - SQL이나 통신 프로토콜 등을 개발할 때 사용. <br>
4. 반복자( Iterator )
    - 자료구조처럼 접근이 잦은 객체에 동일한 인터페이스를 사용하게 만드는 패턴. <br>
5.  중재자( Mediator )
    - 수 많은 객체들간의 복잡한 상호작용(Interface)을 캡슐화하여 객체로정의하는 패턴. <br>
    - 객체사이 의존성을 줄여 결합도를 감소시킬 수 있다. <br>
6. 메멘토( Memento )
    - 특정 시점의 객체 내부 상태를 객체화 함, 이후 요청에따라 객체를 해당 시점의 상태로 돌릴 수 있는 기능을 제공. <br>
    - Ctrl Z와 같은 기능 개발할 때 사용. <br>
7. 옵서버( Observer ) 
    - 한 객체 상태가 변화하면 객체에 상속되어 있는 다른 객체들에게 변화된 상태를 전달하는 패턴. <br>
    - 일대다의 의존성을 정의함. <br>
    - 주로 분산된 시스템 간에 이벤트를 생성/발행(publish)하고 이를 수신(Subscribe)해야 할 때 이용. <br>
8. 상태( State ) 
    - 객체 상태에 따라 동일한 동작을 다르게 처리해야 할 때 사용하는 패턴. <br>
9. 전략( Strategy ) 
    - 동일 계열의 알고리즘을 개별적으로 캡슐화하여 상호교환할 수 있게 만드는 패턴. <br>
10. 템플릿 메소드( Template Method ) 
    - 상위 클래스에서 골격을 정의하고, 하위클래스에서 세부 처리를 구체화하는 패턴. <br>
    - 코드의 양을 줄여 유지보수 쉽게 해준다. <br>
11. 방문자( Visitor )
    - 각 클래스들의 데이터 구조에서 처리 기능을 분리하여 별도의 클래스로 구성하는 패턴. <br>
    - 분리된 처리 기능은 각 클래스를 방문하여 수행. <br>
